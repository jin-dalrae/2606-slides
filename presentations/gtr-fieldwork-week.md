## The test mixed too many questions and introduced avoidable noise

<p class="slide-ref">Testing Process · mistakes</p>

<div class="cards-2">
  <article><h3>Trust before value</h3><p>We requested sensitive documents before clearly showing what participants would receive.</p></article>
  <article><h3>Facilitator-dependent language</h3><p>Net impact, maturity, projection, and milestones required explanation.</p></article>
  <article><h3>Inconsistent sample values</h3><p>The report and dashboard used different illustrative emissions numbers.</p></article>
  <article><h3>Too much per session</h3><p>Two A/B tests, a dashboard, a storyboard, and sharing questions diluted the evidence.</p></article>
</div>

## Repeated blockers matter more than isolated enthusiasm

<p class="slide-ref">What we learned · about testing</p>

<div class="steps-3">
  <article><span class="steps-3__num">01</span><div><h3>Concept and usability interact</h3><p>If the benefit is unclear, a participant may reject the form before evaluating its interface.</p></div></article>
  <article><span class="steps-3__num">02</span><div><h3>Preference is directional</h3><p>At n=5, A/B reactions suggest what to retest; they do not select a universal winner.</p></div></article>
  <article><span class="steps-3__num">03</span><div><h3>Evidence types should stay separate</h3><p>Founder, technical, investment, peer-design, and accessibility feedback answer different questions.</p></div></article>
</div>

---

## Stakeholders put trust and comprehension ahead of features

<p class="slide-ref">What we learned · stakeholders</p>

| Repeated signal | Implication |
| --- | --- |
| Sensitive documents need a clear return and trusted context | Show value first; avoid default upload |
| Product purpose and audience were unclear | State who it is for and preview the output |
| Visual report was easier to interpret than dense text or dashboard | Make the assessment the first-read artifact |
| Generic inputs did not fit every startup | Tailor questions by business model |
| Public sharing depends on the result | Default private; make sharing selective |
| Labels and hierarchy required explanation | Fix comprehension and accessibility before adding features |

---

## Nothing is fully validated yet

<p class="slide-ref">What we learned · ideas to carry forward</p>

<div class="cards-2">
  <article><h3>Short, visible-progress intake</h3><p>Supported enough to compare again with a larger target sample.</p></article>
  <article><h3>Preset, model-specific inputs</h3><p>Carry forward; test whether the defaults reduce effort without hiding important detail.</p></article>
  <article><h3>Separate footprint and handprint</h3><p>Carry forward only with plain-language definitions and transparent calculation logic.</p></article>
  <article><h3>Visual report + explanation</h3><p>The clearest first-read direction; still needs comprehension testing without facilitation.</p></article>
</div>

<p class="slide-closing">“Supported enough to retest” is the correct claim at this stage.</p>

## The product direction becomes narrower and more trust-aware

<p class="slide-ref">Changes &amp; Future Direction · pivots</p>

| Before | After feedback |
| --- | --- |
| Ask for documents during intake | Manual aggregate fields first; files only in a trusted path |
| Explain value at the end | Lead with benefit and preview the output |
| Text-heavy report or dense dashboard | Visual report first; dashboard for follow-up use |
| Generic activity checklist | Business-model-specific questions and defaults |
| Horizontal dashboard cards | Vertical hierarchy with goals and milestones grouped |
| Public-share prompt | Private by default; selective sharing |

## Next round: verify, narrow, then test

<p class="slide-ref">Changes &amp; Future Direction · next steps</p>

<div class="steps-3">
  <article><span class="steps-3__num">01</span><div><h3>Restore the evidence trail</h3><p>Check participant wording against original records and attach notes or transcript to the archive.</p></div></article>
  <article><span class="steps-3__num">02</span><div><h3>Build one focused flow</h3><p>Value proposition → short intake → visual report, with no default document upload.</p></div></article>
  <article><span class="steps-3__num">03</span><div><h3>Run separated studies</h3><p>Test comprehension, trust, and onboarding pattern independently with more recruited startup founders.</p></div></article>
</div>

<p class="slide-closing">Only test real-data workflows after the concept is understandable without facilitator help.</p>
